Muammo: Bir tomonlama autentifikatsiya etarli emasmi?

Klassik Savol-Javob protokolida faqat mijoz o'zini isbotlaydi. Lekin bu juda xavfli:

⚠️ Masalan: Siz "bank.uz" saytiga kirasiz — lekin bu haqiqiy bank serveri ekanligini qanday bilasiz? Tajovuzkor soxta server yaratib, sizning ma'lumotlaringizni o'g'irlashi mumkin.
// Bir tomonlama (unilateral) autentifikatsiya:
C → S : ID_C ← C=mijoz, S=server, ID_C=mijoz identifikatori
S → C : r ← r=server nonce (tasodifiy savol)
C → S : f(K, r) ← f(K,r)=kalit K bilan javob; mijoz o'zini isbotlaydi
S → C : OK
// Lekin: S haqiqiy ekanini kim tekshirdi? HECH KIM!
✅
Yechim: O'zaro (Mutual) Autentifikatsiya

O'zaro autentifikatsiya — ikkala tomon ham bir-birini kriptografik jihatdan tasdiqlaydi:

Alisa → isbotlaydi → Bob
Bob → isbotlaydi → Alisa

Natija: Har ikkalasi ham haqiqiy ekanini biladi!
1

Identifikatsiya

Har ikki tomon o'z identifikatorlarini e'lon qiladi

2

Nonce almashinuvi

Har tomon tasodifiy r_A va r_B nonce yaratadi

3

Ikki tomonlama javob

Har tomon boshqasining nonce-ga javob beradi: f(K, r_B) va f(K, r_A)

4

Ikki tomonlama tekshirish

Har tomon sherigining javobini tekshiradi — ikkalasi tasdiqlangan!

Nonce va Tasodifiylik

O'zaro autentifikatsiyada har ikki tomon o'z nonce-ini yaratadi:

r_A — Alisa tasodifiy soni (har sessiyada yangi)
r_B — Bob tasodifiy soni (har sessiyada yangi)

Nima uchun muhim?
Eski sessiyadan r_A ni qayta yuborsa, hujumchi eski MAC ni ham yuborishi kerak.
Lekin Bob har safar yangi r_B yuboradi → eski MAC yangi r_B ga mos kelmaydi → hujum muvaffaqiyatsiz!
💡 Ikkala nonce birga ishlatilganda — har ikki tomonga oid replay hujumlari ham bloklanadi.
🔐
HMAC — Qisqacha tushuntirish

HMAC (Keyed-Hash Message Authentication Code) — kalit va xabardan maxfiy imzo (MAC) yaratadi. Faqat kalitni bilgan tomon to'g'ri MAC hisoblashi mumkin.

Oddiy g'oya:
MAC = maxfiy_funksiya(Kalit, Xabar) ← K=kalit, xabar=nonce+identifikator

O'zaro autentifikatsiyada:
Bob: MAC = HMAC(K, r_B + r_A + "Alisa") ← r_B,r_A=nonce, "Alisa"=ID_A
Alisa: MAC = HMAC(K, r_A + r_B + "Bob") ← "Bob"=ID_B

Haqiqiy tizimlarda SHA-256 xesh ishlatiladi. Bizda demo uchun soddalashtirilgan versiya.
📐
ISO/IEC 9798-2 Formal Ta'rif
// 3-xabarli o'zaro autentifikatsiya (simmetrik kalit):

Alisa → Bob : r_A ← r_A: Alisa nonce (tasodifiy son)

Bob → Alisa : r_B ‖ MACK(r_B ‖ r_A ‖ ID_A)
↑ ‖ = birlashtirish; MAC_K = kalit K bilan imzo; ID_A = Alisa identifikatori (ismi)
↑ Bob o'zini isbotladi

Alisa → Bob : MACK(r_A ‖ r_B ‖ ID_B)
↑ ID_B = Bob identifikatori (ismi)
↑ Alisa o'zini isbotladi

Tekshirish:
Alisa: MACK(r_B ‖ r_A ‖ ID_A) =? qabul qilingan ← Alisa o'z MAC hisoblab solishtiradi
Bob: MACK(r_A ‖ r_B ‖ ID_B) =? qabul qilingan ← Bob o'z MAC hisoblab solishtiradi
✅ To'g'rilik: Agar K ikkala tomonda bir xil bo'lsa va xabarlar o'zgartirilmagan bo'lsa — ikkalasi ham muvaffaqiyatli tasdiqlaydi.

1
O'zaro autentifikatsiya — tushuncha va zarurligi

O'zaro autentifikatsiya — ikkala tomon ham bir-birini kriptografik jihatdan tasdiqlaydi. Bir tomonlama autentifikatsiyada faqat mijoz o'zini isbotlaydi, lekin server haqiqiy ekanini tekshirilmaydi (soxta server xavfi).

Alisa → isbotlaydi → Bob
Bob → isbotlaydi → Alisa
Natija: Har ikkalasi ham haqiqiy ekanini biladi!

Asosiy bosqichlar: ① Identifikatsiya ② Nonce almashinuvi ③ Ikki tomonlama MAC javob ④ Tekshirish

▼
2
Nonce va HMAC

Nonce — har sessiyada yangi tasodifiy son. Eski nonce qayta ishlatilsa, MAC mos kelmaydi → replay hujum bloklanadi.

r_A — Alisa nonce (har safar yangi)
r_B — Bob nonce (har safar yangi)
Ikkalasi ham MAC ga kiritiladi → ikkala tomonga replay himoya

HMAC — kalit va xabardan maxfiy imzo (MAC) yaratadi. Faqat kalitni bilgan tomon to'g'ri MAC hisoblashi mumkin.

▼
3
ISO/IEC 9798-2 — 3-xabarli protokol

Simmetrik kalit K_AB — oldindan kelishilgan. 3 ta xabar:

① A → B : r_A (r_A=nonce)
② B → A : r_B ‖ MAC_K(r_B ‖ r_A ‖ ID_A) (‖=birlashtirish, ID_A=Alisa ismi)
③ A → B : MAC_K(r_A ‖ r_B ‖ ID_B) (ID_B=Bob ismi)

Tekshirish: Alisa MAC_B ni, Bob MAC_A ni tekshiradi.

Himoya: ID_A, ID_B MAC-ga kiritiladi → reflection hujum bloklanadi.

▼
4
Needham-Schroeder va KDC

KDC orqali. 5 ta xabar. Lowe hujumi (1995) — tuzatish: NSL da ID_B qo'shiladi.

① A → KDC : ID_A ‖ ID_B ‖ r_A (KDC=kalit markazi)
② KDC → A : E_K_A(K_S ‖ ... ‖ E_K_B(K_S ‖ ID_A)) (E=shifrlash, K_S=sessiya kaliti)
③ A → B : E_K_B(K_S ‖ ID_A ‖ ID_B) ← NSL tuzatmasi
④ B → A : E_K_S(r_B) (r_B=Bob nonce)
⑤ A → B : E_K_S(r_B - 1) (Alisa tasdiqi)
▼
5
Kerberos v5

AS + TGS. Timestamp ±5 daqiqa replay himoyasi. TGT → Service Ticket → Xizmat.

C → AS : ID_C ‖ ID_TGS ‖ T_S (C=mijoz, T_S=vaqt)
AS → C : E_K_C(K_C,TGS) ‖ TGT (TGT=bilet)
C → TGS : ID_S ‖ TGT ‖ Auth (Auth=autentifikator)
TGS → C : E_K_C,TGS(K_C,S) ‖ Ticket_S (Ticket_S=xizmat bileti)
C → S : Ticket_S ‖ Auth_2
S → C : E_K_C,S(T_C+1) ← o'zaro tasdiq
▼
6
STS (Station-to-Station)

DH + raqamli imzo. KDC kerak emas. Forward Secrecy — har sessiyada yangi x,y.

① A → B : g^x mod p (x=sir son, p=tub son)
② B → A : g^y ‖ E_K(Sign_sk_B(g^y ‖ g^x)) (sk_B=Bob xususiy kalit, K=g^xy)
③ A → B : E_K(Sign_sk_A(g^x ‖ g^y)) (sk_A=Alisa xususiy kalit)
K = g^xy mod p — sessiya kaliti

ISO/IEC 9798-2 — Simmetrik Kalit, 3-xabar

Eng sodda va keng tarqalgan o'zaro autentifikatsiya standarti. Simmetrik kalit K_{AB} — oldindan kelishib olingan:

K_{AB} biladi
→ r_A ① Alisa nonce yuboradi
← r_B ‖ MAC(K, r_B‖r_A‖ID_A) ② Bob: nonce + MAC
→ MAC(K, r_A‖r_B‖ID_B) ③ Alisa: MAC javob
👨
Bob (B)
K_{AB} biladi
// Alisa xabari ① : A → B : r_A ← r_A: Alisa nonce (tasodifiy son)

// Bob xabari ② : B → A : r_B ‖ MACK(r_B ‖ r_A ‖ ID_A)
↑ ‖ birlashtirish; MAC_K = K bilan imzo; ID_A = Alisa identifikatori

// Alisa tekshiradi ② : Alisa: MAC tekshirildi — mos keladimi? → Bob tasdiqlandi ✓

// Alisa xabari ③ : A → B : MACK(r_A ‖ r_B ‖ ID_B) ← ID_B = Bob identifikatori

// Bob tekshiradi ③ : Bob: MAC tekshirildi — mos keladimi? → Alisa tasdiqlandi ✓
✅ Nima uchun xavfsiz? r_A va r_B ikkisi ham MACga kiritiladi. Eva eski xabarni qayta yuborsa, Bob yangi r_B beradi → eski MAC mos kelmaydi. Kalitni bilmasa, yangi MAC hisoblab bo'lmaydi.

Needham-Schroeder Protokoli (1978)

Birinchi rasmiy o'zaro autentifikatsiya protokoli. KDC (Key Distribution Center) — ishonchli uchinchi tomon orqali ishlaydi:

👩
Alisa
K_A biladi
→ID_A, ID_B, r_A① KDC ga so'rov
←E_{K_A}(K_S, ID_B, r_A, T_B)② Sessiya kalit + ticket
→T_B = E_{K_B}(K_S, ID_A)③ Bob uchun ticket
←E_{K_S}(r_B)④ Bob nonce
→E_{K_S}(r_B - 1)⑤ Alisa javobi
🏛️
KDC
K_A, K_B biladi
// Izoh: E_K = K bilan shifrlash; K_S = sessiya kaliti; ID_A/ID_B = identifikatorlar
① A → KDC : ID_A ‖ ID_B ‖ r_A ← Alisa kim bilan aloqa qilmoqchi
② KDC → A : EK_A(K_S ‖ ID_B ‖ r_A ‖ EK_B(K_S ‖ ID_A))
↑ K_S — yangi sessiya kaliti; E_K_B(...) — Bob uchun ticket (bilet)
③ A → B : EK_B(K_S ‖ ID_A) ← ticket: faqat Bob ocha oladi
④ B → A : EK_S(r_B) ← Bob nonce, K_S bilan shifrlangan
⑤ A → B : EK_S(r_B - 1) ← Alisa r_B ni qaytardi (tasdiq)
⚠️ Lowe hujumi (1995): Eva eski sessiya kalitini qayta ishlatib, Bobni aldashi mumkin! Bu kamchilikni Lowe topdi va protokol NSL (Needham-Schroeder-Lowe) versiyasida tuzatildi.
// NSL tuzatmasi — ③-xabarga ID_B qo'shildi:
③ A → B : EK_B(K_S ‖ ID_A ‖ ID_B) ← Lowe tuzatmasi

Kerberos v5 — KDC Asosida Amaliy Protokol

MIT tomonidan ishlab chiqilgan. Timestamp asosida replay himoyasi. Windows Active Directory standart protokoli:

// Izoh: C=mijoz, S=xizmat, AS=autentifikatsiya serveri, TGS=bilet beruvchi
// TGT=bilet, Auth=autentifikator, T_S/T_C=timestamp (vaqt)

// 1-bosqich: TGT olish (AS bilan)
C → AS : ID_C ‖ ID_TGS ‖ T_S ← mijoz, TGS, vaqt
AS → C : EK_C(K_{C,TGS}) ‖ TGT ← sessiya kaliti + TGT bilet

// 2-bosqich: Xizmat bileti olish (TGS bilan)
C → TGS : ID_S ‖ TGT ‖ Auth ← qaysi xizmat, TGT, Auth
TGS → C : EK_{C,TGS}(K_{C,S}) ‖ Ticket_S ← xizmat sessiya kaliti + bilet

// 3-bosqich: Xizmatga kirish
C → S : Ticket_S ‖ Auth_2
S → C : EK_{C,S}(T_C + 1) ← o'zaro tasdiq (S ham o'zini isbotladi)

STS (Station-to-Station) Protokoli

DH kalit almashinuvini raqamli imzo bilan birlashtiradi. KDC siz ishlaydi, Forward Secrecy ta'minlaydi:

👩
Alisa
sk_A, pk_A (xususiy va ochiq kalit)
→g^x mod p① DH qism
←g^y, E_K(σ_B)② DH + imzo
→E_K(σ_A)③ Alisa imzosi
👨
Bob
sk_B, pk_B (xususiy va ochiq kalit)

Qisqacha: Alisa va Bob har biri sir son (x, y) tanlaydi. gx va gy almashadi. Ikkalasi ham bir xil K = gxy ni hisoblaydi. Imzolar MITM hujumini bloklaydi.

// Izoh: g^x = g sonning x darajasi (mod p); x,y = sir sonlar; K = sessiya kaliti
① A → B : gx ← Alisa sir son x tanladi, g^x ochiq yuboradi
② B → A : gy + imzo σ_B ← Bob g^y + imzo; K = g^xy ikkalasida ham bir xil
③ A → B : imzo σ_A ← Alisa imzosi

Tekshirish: Har tomon sherigining imzosini tekshiradi (Verify) → ikkalasi tasdiqlanadi
✅ Forward Secrecy: Har sessiyada yangi x va y tanlangan — eski sessiya kalitlarini hech kim tiklay olmaydi, hatto xususiy kalitlar keyinroq fosh bo'lsa ham.

Reflection Attack (Aks hujum)

O'zaro autentifikatsiya protokolida nonce-lar to'g'ri birlashtirilmasa Eva sheriga o'z savoliga o'zi javob berdirib tasdiqlata olishi mumkin:

// Zaif protokol (noto'g'ri):
① A → B : r_A (r_A=Alisa nonce)
② B → A : r_B ‖ MAC(K, r_A) ← Bob isboti; zaif: ID yo'q!
③ A → B : MAC(K, r_B) ← Alisa isboti

// Reflection hujum — Eva Alisa o'rnida:
Eva → Bob : r_1
Bob → Eva : r_2 ‖ MAC(K, r_1)
Eva ikkinchi kanal ochadi → Bob : r_2
Bob → Eva : r_3 ‖ MAC(K, r_2)
Eva → Bob : MAC(K, r_2) ← Bobdan oldi!
ISO 9798-2 himoyasi: ID_A va ID_B MAC ichiga kiritiladi
✅ Himoya: MAC(K, r_B ‖ r_A ‖ ID_A) — ID_A (Alisa ismi) va ID_B (Bob ismi) ni qo'shish reflection hujumini bloklab qo'yadi, chunki Eva ularni almashtirib qo'ya olmaydi.
🎭
Lowe Hujumi (Needham-Schroeder ga)
// Original NS (zaif) — Eva server sifatida:
// E_K=shifrlash, K_S=sessiya kaliti, ID_A=Alisa identifikatori
① A → Eva : EK_B(K_S ‖ ID_A) ← Alisa Evani Bob deb o'ylaydi
Eva → B : EK_B(K_S ‖ ID_A) ← ticket ni Bob ga uzatadi

Eva → B : EK_S(r_B - 1)

// NSL tuzatmasi:
B → A : EK_S(r_B ‖ ID_B) ← ID_B qo'shildi → Lowe hujumi bloklandi
